## 背景

Claw 当前方向是 OpenClaw 式电脑接管智能体：iPhone/iPad 端负责规划、风险展示、审批、envelope 发送和事件查看，真实电脑操作发生在桌面 Gateway。Gateway 已输出多类 metadata-only artifact 复核摘要，手机端已经能在 Mission Run、Gateway session card 和 result row 中展示。

随着复核类型增多，Mission Run 首屏出现两个问题：

1. 用户需要人工扫完整组 review row，才能判断当前最该处理的是审批、阻断、metadata 缺失、文件写入、Shell、最终提交、Replay Guard 还是 agent loop handoff。
2. iPad/mac 宽屏工作台虽然复用了 Mission Run 面板，但首屏缺少按风险和可行动性排序的紧凑队列，不能像电脑智能体控制台那样快速提示下一步复核重点。

本轮要新增 Mission Run 复核优先队列：只基于现有 `ClawMissionRunSummary`、现有 review summary 和 session/action 状态派生，不读取 Gateway `file://` payload，不新增 Gateway 协议字段，不把任何 raw URL/path/command/diff/stdout/stderr/token/header/toolArguments 带入 UI。

## 目标

- 新增 presentation model，例如 `ClawMissionRunReviewPriorityItem`，用于表达首屏复核队列项。
- `ClawMissionRunSummary` 增加 `reviewPriorityQueue: [ClawMissionRunReviewPriorityItem]` 或等价字段。
- 队列从既有 Mission Run summary 输入派生，覆盖：
  - 用户审批/最终提交闸门。
  - Gateway blocked / failed / retryable action。
  - Replay Guard 重复任务跳过。
  - File Change Safety、Shell Command Safety、Delivery Safety 等高风险执行面。
  - Browser Control、Extraction completeness、Accessibility、Gateway capability、Artifact metadata、AgentTrace 等复核摘要。
  - metadata 缺失时的“metadata 待同步”降级项。
- Mission Run 首屏在 artifact chips 后、详细 review rows 前展示优先队列，compact 与 iPad regular 工作台自动复用。
- Gateway session card 和 result row 保持详细 review row，不在 view 层重复排序逻辑。
- 增加 XCTest 和 Swift logic smoke，断言队列顺序、空队列、metadata 缺失降级、敏感 metadata 不外显。
- 同步 README、协议、flow/flowchart、测试说明和 update_log。

## 非目标

- 不新增 action kind、event kind、artifact kind、schema 字段、envelope 字段或 Gateway API。
- 不改 Gateway 执行权限、allowlist、Shell 策略、浏览器策略、文件 workspace 策略或最终提交闸门。
- 不读取 artifact `reference`，不打开 `file://` payload，不展示 raw artifact 内容。
- 不在 UI 或 metadata 中展示 raw URL、search query、HTML/page text、path、workspace/sessionWorkspace、文件名/文件内容/diff/patch、raw command、binary/args、cwd、stdout/stderr、草稿正文、paste text、按键原文、token、Authorization/header、cookie、自然语言 instruction 或 `toolArguments`。
- 不把优先队列描述成安全保证、恶意检测、完整审计、自动修复器或跨进程 exactly-once。
- 不把排序逻辑只写在 SwiftUI View 中；队列必须能被 LogicSmoke/XCTest 直接断言。
- 不删除现有详细 review rows，不用少展示来掩盖风险。

## 队列设计建议

新增模型字段建议：

```swift
struct ClawMissionRunReviewPriorityItem: Identifiable, Equatable, Codable, Sendable {
    var id: String
    var rank: Int
    var severity: ClawMissionRunReviewPrioritySeverity
    var title: String
    var status: String
    var reason: String
    var icon: String
    var reviewKind: String
    var actionHint: String
    var isActionable: Bool
    var hasMetadata: Bool
}
```

severity 可用 enum：`critical` / `high` / `medium` / `low` / `info`。所有 `title/status/reason/actionHint/reviewKind` 必须来自固定文案或已脱敏 summary 字段，不接受任意 metadata key/value。

建议排序规则：

1. `critical`：Gateway blocked、action failed、Replay Guard duplicate skip、最终提交/用户确认闸门、路径逃逸或 shell 执行异常等必须人工处理事项。
2. `high`：File Change Safety、Shell Command Safety、Delivery Safety，尤其 metadata 缺失、policy blocked、write failed、submit blocked、execution attempted/executed 等状态。
3. `medium`：Browser Control、Extraction completeness、Accessibility、Gateway capability 中影响任务判断的状态，如 network blocked、incomplete extraction、capability unavailable、accessibility unavailable。
4. `low`：Artifact metadata coverage、AgentTrace handoff/missing signals、一般 redaction/metadata present 状态。
5. `info`：纯成功、完整且无阻断的复核摘要。

队列最多展示 5 项；模型可保留 5 项或更多，但 UI 只展示前 5 项。排序必须稳定，同 rank 内按固定 review kind 顺序输出，避免每次刷新抖动。

建议固定 review kind 顺序：

`approval`, `gateway-status`, `replay-guard`, `delivery-safety`, `shell-safety`, `file-change-safety`, `browser-control`, `extraction-completeness`, `accessibility`, `gateway-capability`, `artifact-metadata`, `agent-trace`。

metadata 缺失策略：

- 如果某类 review 只有 legacy fallback，队列项 `hasMetadata=false`，status/reason 明确“metadata 待同步”。
- 不能把 metadata 缺失误判为安全、已阻断或已执行。

## 风险与禁止项

- 风险：排序规则过于主观。用固定 severity/rank/review kind order 控制稳定性，并用测试锁定。
- 风险：队列文案误导为安全保证。文案必须表达“待复核/待确认/metadata 待同步”，不要写“绝对安全”。
- 风险：重复 UI 逻辑。排序和文案尽量放模型/派生层，View 只展示。
- 禁止把项目方向改成法律/法务产品。
- 禁止自然语言直连 Shell、AppleScript 或桌面动作。
- 禁止绕过 allowlist、审批闸门、workspace 限制和 artifact 审计。
- 禁止把旧 artifact、旧 output 或本地输出冒充最新云端结果。
