# v0.41 / Mac Agent 下一步候选卡组

## 版本号和版本分配依据

- 版本号：v0.41。
- 版本分配依据：v0.40 已完成 Mac Agent Evidence Coverage Map，把 action、artifact、metadata、payload 边界和人工复核状态串成证据覆盖视图。下一步应继续增强 mac/iPad 工作台的“智能体可用性”，把已有就绪、预检、证据、审批和 loop 交接信息整理为可扫视的下一步候选卡组，但仍保持 presentation-layer、人工审批和 Gateway 权限边界。

## 背景

Claw 当前已经能在 Mission Run 中展示 Operator Strip、Loop 继续态势、Mac Agent Readiness Board、Mac Gateway Action Preflight Matrix、Mac Agent Evidence Coverage Map、Focus Context、Review Detail Dock、Review Trail、Approval Queue、Payload Safety Ledger、Artifact Evidence Index、复核态势和下一步复核行动。

这些组件分别回答“桌面端是否就绪”“action 是否可派发”“证据是否覆盖”“payload 是否安全”“当前该复核哪一项”。但用户在 mac/iPad 宽屏工作台中仍需要跨多个组件判断下一步候选：是先审批、先等待 metadata、先看某类证据、先处理失败/重试，还是在 AgentTrace 可继续时由用户显式触发下一轮。

本轮选择一个保守小目标：新增手机端 presentation-layer 的 Mac Agent Next Step Deck，用固定候选卡把“下一步可做什么”汇总出来。它不是自动计划器，不生成新 envelope，不自动继续，不新增真实权限。

## 目标

新增 `Mac Agent 下一步候选卡组`：

- 从既有 `ClawMissionRunSummary`、`nextReviewAction(focusedOn:)`、`loopContinuationSummary(focusedOn:)`、`macAgentReadinessBoard(focusedOn:)`、`macGatewayActionPreflightMatrix(focusedOn:)`、`macAgentEvidenceCoverageMap(focusedOn:)`、`approvalQueueSummary(focusedOn:)`、`reviewReadinessSummary(focusedOn:)` 和现有计数派生。
- 新增轻量模型，例如：
  - `ClawMissionRunNextStepCandidate`
  - `ClawMissionRunNextStepDeck`
- 候选卡固定覆盖 4 到 6 类即可，建议保守采用：
  - 人工确认：审批、Gateway 等待确认、最终提交闸门、AgentTrace handoff。
  - 证据补齐：metadata 待同步、artifact/coverage 未覆盖、观察/能力降级。
  - 失败复核：阻断、失败、可重试。
  - Loop 下一步：AgentTrace 可继续、需要证据、完成或阻断。
  - 抽查复核：无阻断时指向最高优先 detail review 或 next review action。
- 每张候选卡只包含固定 title/status/guidance/icon/tone/reviewKind/reviewTitle/count/flag，不读取 payload、artifact reference 或 action `toolArguments` 原文。
- 在 compact Mission Run 中展示，建议放在 Evidence Coverage Map 后、指标网格前；在 iPad/mac regular Review Detail Dock 中展示，建议放在 Evidence Coverage Map 后、Review Trail 前。
- 卡片点击只能改变本地 `focusedReviewKind`，不执行 Gateway 动作、不审批、不发送、不重试、不自动继续。
- UI 文案要明确“候选/建议/聚焦”，避免声称自动接管、自动执行或自动安全裁决。

## 非目标

- 不新增 Gateway action、artifact kind、schema 字段、event kind 或 WebSocket 协议。
- 不实现真实多轮 agent loop、完整 macOS Accessibility bridge、Playwright/browser-use 兼容控制器、后台保活、真实心跳、配对或完整 artifact payload viewer。
- 不把候选卡写入 `ClawMobileEnvelope`，不回传 Gateway，不改变 planner/bridge 生成的 `toolArguments`。
- 不做自动审批、自动重试、自动发送、自动继续、自动安全裁决或自动执行授权。
- 不读取或展示 Gateway `file://` payload、artifact `reference`、文件内容、命令输出、diff、网页正文、草稿正文、raw URL/path/token/header/Authorization 或 action `toolArguments` 原文。

## 实现步骤

1. 在 `Claw/Core/ClawModels.swift` 新增候选卡模型：
   - `ClawMissionRunNextStepCandidate`
   - `ClawMissionRunNextStepDeck`
   - 字段建议包含 `id`、`rank`、`title`、`status`、`guidance`、`icon`、`tone`、`reviewKind`、`reviewTitle`、`canFocusReview`、`isFocused`、`requiresHumanAction`、`hasMetadataGap`、`isBlocked`、`isRetryable`、`canContinueLoop`。
2. 在 `ClawMissionRunSummary` extension 新增：
   - `var macAgentNextStepDeck: ClawMissionRunNextStepDeck`
   - `func macAgentNextStepDeck(focusedOn reviewKind: String?) -> ClawMissionRunNextStepDeck`
   - 私有 helper，例如 `nextStepCandidates(...)`、`nextStepCandidate(...)`，避免大闭包造成 Swift type-check 压力。
3. Deck 派生规则建议：
   - idle：`isReviewable == false`，不虚构候选 row，status 指向“尚无 Gateway 证据/生成任务后显示候选”。
   - pre-send：候选优先显示手机审批/人工确认，允许聚焦 `approval`；不能伪装已有 Gateway evidence。
   - sent 且有失败/阻断/可重试：优先给失败复核候选，reviewKind 可用 `gateway-status` 或对应 detail review。
   - approval queue 有可行动项：给人工确认候选，reviewKind 使用 queue primary/focused item。
   - Evidence Coverage 或 Readiness 有 metadata gap：给证据补齐候选，reviewKind 使用对应 primary item 或 next review action。
   - AgentTrace/Loop 可继续：给 Loop 下一步候选，但文案必须写“用户显式触发下一轮”，不能自动继续。
   - 无阻断时：给抽查复核候选，指向 `nextReviewAction` 或最高优先 detail review。
4. 聚焦规则：
   - 使用 `activeReviewFocus(from:)` 过滤未知聚焦。
   - `approval`、`gateway-status` 等状态项可作为候选聚焦，但 Detail Rows 保持既有全量 fallback。
   - 未知/过期聚焦时 Deck 不保留焦点，不崩溃，不隐藏可用候选。
5. UI：
   - 在 `Claw/Views/ContentView.swift` 新增 `ClawMissionMacAgentNextStepDeckView` 和 `ClawMissionMacAgentNextStepCandidateRow`。
   - compact Mission Run：在 `ClawMissionMacAgentEvidenceCoverageMapView` 后、指标网格前插入 Deck。
   - regular Review Detail Dock：在 Coverage Map 后、Review Trail 前插入 Deck。
   - 行按钮只调用 `onFocusReviewKind(reviewKind)`；accessibility hint 必须说明只聚焦，不执行 Gateway 动作、不审批、不发送、不重试、不自动继续。
   - 保持 8px 左右圆角风格，与现有 board/matrix/map 一致；不要做新的 landing/hero/装饰图。
6. 测试：
   - `Tools/LogicSmoke.swift` 增加 Deck 派生断言，覆盖 idle、pre-send、sent、focus、status-only focus、stale focus、Shell 高风险/可重试、Loop 可继续和敏感字符串不外显。
   - `ClawTests/ClawTests.swift` 增加同类 XCTest。
   - 敏感字符串集合沿用既有禁止项，至少包含 `Authorization`、`Bearer`、`toolArguments`、`file://`、`/private`、`/Users`、`/home`、`C:\\`、`stdout`、`stderr`、`diff`。
   - 注意拆分长数组、长字符串和复杂表达式，避免 Swift 编译器 type-check 超时。
7. 文档：
   - 同步 `README.md`、`Docs/claw-mobile-gateway-protocol.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和 `update_log.md`。
   - 文档必须明确 Deck 是 presentation-layer 候选卡组，不写入 envelope，不回传 Gateway，不读取 payload，不扩大权限。

## 验收标准

- idle 时 Deck 不可复核、不虚构候选 row、不提供 primary focus。
- pre-send 时 Deck 能显示手机审批/人工确认候选，可聚焦 `approval`，但不伪装 Gateway evidence、metadata 或 artifact 已覆盖。
- sent 后 Deck 能按固定优先级汇总人工确认、证据补齐、失败复核、Loop 下一步和抽查复核候选。
- `delivery-safety`、`shell-safety`、`agent-trace` 等 detail review 聚焦时，Deck 能标记对应候选；`gateway-status` / `approval` 状态项聚焦时不破坏 detail fallback；未知聚焦回退 nil。
- Shell 高风险/可重试路径能把候选指向 Shell 安全或 Gateway 状态复核，但不外显命令、stdout/stderr、cwd、path、diff 或 `toolArguments`。
- AgentTrace 可继续路径可以显示“用户显式触发下一轮”的候选，但不能自动调用继续、发送、审批或重试。
- UI row 点击只改变本地聚焦，不触发 Gateway 动作、不审批、不发送、不重试、不自动继续。
- 新增可见字符串和测试收集的 visible chunks 不包含 raw URL/path/command/stdout/stderr/diff/token/header/Authorization/Bearer/草稿正文/`toolArguments`/`file://`。
- 文档完整同步，且没有把项目方向改成法律/法务/合同/诉讼垂直产品。
- GitHub Actions 最新 `origin/main` run 通过，Agent C 下载并复判未加密 artifact 通过。

## 风险和禁止项

- 风险：Deck 容易被误解为自动执行计划。所有文案必须使用“候选”“建议”“聚焦”“用户显式触发”，不得暗示自动执行或自动授权。
- 风险：候选优先级可能与既有 `nextReviewAction`、Approval Queue 或 Evidence Coverage 重复。应复用既有摘要并保持小范围，不重写排序体系。
- 风险：Swift summary extension 已较大。实现时拆 helper，避免复杂嵌套闭包和长表达式。
- 禁止新增真实权限、Gateway action、artifact kind、event kind、schema 字段或 WebSocket 协议。
- 禁止读取或解析 artifact `reference`、Gateway `file://` payload、文件内容、命令输出、diff、网页正文、草稿正文或 action `toolArguments` 原文。
- 禁止把 raw token、Authorization header、Bearer、完整 workspace path、raw URL、raw command、stdout/stderr、draft body 或 `toolArguments` 放入 UI、日志、metadata、测试 visible string 或文档示例。
- 禁止绕过 allowlist、审批闸门、workspace 限制、replay guard 或 token 指纹规则。
- 禁止删除测试断言来制造通过。
- 禁止混入无关重构、格式化或其他 Agent/用户改动。
